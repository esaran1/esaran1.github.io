Quant Options Side project

Building VolSurf and Getting Into Quant

February 2026 · 5 min read

I’ve been spending more time on quant-style projects lately—market structure, options, that kind of thing. The limit order book simulator was one piece of it: you get to build the “engine” and then train or test agents on top. VolSurf is the other side: instead of simulating how orders flow, you’re looking at how the options market is already pricing volatility. Pick a ticker, pull the chain, back out implied vol, and you get this surface—strike on one axis, time to expiry on the other, and the height is basically “how much uncertainty is the market putting in this option?” I wanted that in one place, with no external pricing API. So I built it.

What I actually built

VolSurf is a small full-stack app: FastAPI backend, React frontend. You choose a ticker (default SPY), set risk-free rate and div yield, and the backend grabs option chains from Yahoo Finance, runs everything through Black–Scholes with dividends, and solves for IV so the model price matches the market. Then it sends back the surface plus summary stats, a smile slice, term structure, and a put–call parity check. The frontend has a 3D view you can flip between IV and the Greeks (Delta, Gamma, Theta), so you see the same grid in different “modes.” I also added a simple smile-vs-smirk read—basically “is the curve more symmetric or is one side (usually puts for indices) carrying more vol?”—and a table + CSV export so you can poke at the numbers.

I didn’t use any external option-pricing library. Implementing Black–Scholes and the Greeks was straightforward; the annoying part was the IV solver. Newton–Raphson is fast when it works, but for deep in- or out-of-the-money options it can blow up or converge to the wrong thing. So I added a bisection fallback: if Newton doesn’t behave, we bracket the vol and bisect. It’s slower but stable. Took a bit of tuning (bounds, max iterations) but now the surface looks sane across the grid.

Why I’m into this stuff

Quant work has been a lot of fun so far. The math is satisfying—you can check your work. Put–call parity is a great sanity check: if market call minus put doesn’t line up with the theoretical no-arbitrage value, something’s off (data, timing, or the model). Same with the shape of the surface: indices usually show put skew, and you can see it in the app. So you’re not just writing code; you’re implementing ideas that have clear, testable implications.

The LOB project was more about execution and market microstructure; VolSurf is about how volatility is already encoded in prices. They complement each other. I’m not sure what I’ll do next—maybe hook the two worlds together in some way, or add more underlyings and a better data source—but I’m definitely going to keep building in this space.
