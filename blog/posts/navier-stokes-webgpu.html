Building a 2D Fluid Sim in the Browser with WebGPU

February 2026 · 6 min read

I wanted to do a real-time fluid simulation that runs in the browser and actually uses the GPU—no canvas 2D or a big 3D engine. WebGPU is still not everywhere (Safari’s getting there, Chrome and Edge have had it for a while), but when it’s there you get compute shaders and render pipelines without pulling in Three.js or Unity. So I built a 2D incompressible Navier–Stokes sim using Stam’s Stable Fluids algorithm, all in WGSL and the raw WebGPU API. You get a full-screen canvas; you click-drag to inject dye and velocity. That’s it. This post is about what that actually took.

Why no Three.js

I could have used a library that wraps WebGPU or WebGL, but I wanted to see the whole pipeline: request adapter and device, create storage textures and uniform buffers, compile WGSL, build compute and render pipelines, and record a command buffer every frame. You learn more when the “engine” is just your own code. The downside is you hit all the little WebGPU quirks yourself—like not being able to read and write the same texture in one compute pass, so you need double buffers (ping-pong) for velocity, density, and pressure. Once that pattern clicks, it’s the same everywhere: run the pass, swap read/write, run the next.

The pipeline

Each frame we advect velocity (semi-Lagrangian: trace backward, sample), then advect density with a bit of dissipation so the dye fades. If the mouse is down we splat a Gaussian blob of velocity and color at the cursor—that’s what makes the drag feel responsive. Then we compute divergence, solve for pressure with 20 Jacobi iterations (the slow part, but 20 is enough for a 512×512 grid to look good), and subtract the pressure gradient so the velocity is divergence-free. Finally we draw a full-screen triangle and sample the density texture in the fragment shader. All of that runs in one command encoder per frame; we submit a single command buffer. I use R32Float for pressure and divergence because some devices are picky about R16Float storage; velocity and density are RGBA16Float (only .rg for velocity).

The shaders are plain WGSL files loaded with Vite’s ?raw and concatenated with a small common block (SimParams struct, grid size, etc.). No fancy build step. Parameters—time step, dissipation, splat radius, mouse position and delta—go into one uniform buffer that we update every frame (and again right before the splat pass when the mouse is down).

What was fun

Getting the first frame of “smoke” to move correctly felt great. So did the first time the splat actually pushed the fluid in the direction you dragged. WebGPU’s error messages are still not always helpful, so a lot of it was adding labels to bind groups and pipelines and stepping through. ResizeObserver on the canvas so we don’t try to get a GPU context at zero size, and a small landing overlay that shows WebGPU status (“Checking…”, “Ready”, or the error) before you click into the sim—those small UX bits made it feel like a real app instead of a demo. I didn’t add velocity diffusion yet (viscosity is in the params but not wired to a solve); advection is nearest-neighbor, so you could improve the look with bilinear or higher-order sampling. For a first pass it’s enough. If you’re curious about the full layout and how to run it, I wrote up the project here.
