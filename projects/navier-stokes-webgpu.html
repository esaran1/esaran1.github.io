WebGPU Graphics Simulation

Navier–Stokes WebGPU

Real-time 2D incompressible fluid simulation in the browser. Runs on the GPU via the raw WebGPU API—no Three.js or other 3D libs. You click-drag on a full-screen canvas to inject dye and velocity; a landing page explains the app and lets you drop into the sim.

What it is

The simulation uses Stam’s Stable Fluids algorithm: advect velocity and density (dye), optionally splat a Gaussian blob of velocity and color at the mouse, compute divergence, solve the pressure Poisson equation with Jacobi iterations, then subtract the pressure gradient so the field is (approximately) divergence-free. Every step runs in compute shaders; the final frame is rendered by sampling the density texture in a full-screen fragment shader. The grid is fixed at 512×512 and stretched to the viewport. Velocity and density live in RGBA16Float textures (only .rg for velocity); pressure and divergence use R32Float so storage works across more devices. Because compute shaders can’t read and write the same texture in one pass, each field uses a double buffer—ping-pong—and we swap read/write after each pass. One command encoder per frame runs all the compute passes and the render pass, then submits a single command buffer.

Tech stack

  • App — React 19 + TypeScript, Vite 7
  • Graphics — WebGPU (adapter/device, compute + render pipelines)
  • Shaders — WGSL, raw strings loaded via Vite ?raw
  • UI — Tailwind (landing, overlay, controls)
  • Types — @webgpu/types

No game engines or GPU wrappers—simulation and rendering are implemented directly on WebGPU.

Per-frame pipeline

Each frame: advect velocity (semi-Lagrangian, trace back and sample), then advect density with dissipation < 1 so the dye fades. If the mouse is down, splat velocity and dye at the cursor. Compute divergence with central differences, solve ∇²p = div(v) with 20 Jacobi iterations (pressure in its own R32Float double buffer), then gradient subtract so the velocity is incompressible. Render by drawing a full-screen triangle and sampling the current density texture. All stencils use clamped indices to avoid underflow. Time step, viscosity, dissipation, resolution, and splat params live in a single SimParams uniform buffer, rewritten each frame.

Main pieces

GPUContext — Singleton that requests the adapter and device once, caches them, and handles missing WebGPU or device loss (uncapturederror).

TextureManager — Creates 2D textures (r32float or rgba16float) with storage and optional sampling bindings.

DoubleBuffer — Two textures per field; exposes read/write and swap() for ping-pong.

FluidSimulator — Builds all compute pipelines (advect, divergence, jacobi, gradient_subtract, splat) and the render pipeline; implements step(params) and manages bind groups.

FluidCanvas — Resizes the canvas (e.g. ResizeObserver), waits for non-zero size, gets the GPU context, creates the simulator, and runs the requestAnimationFrame loop. Tracks mouse position, delta, and mouseDown and passes them into params.

App — Full-viewport canvas plus a landing overlay (title, short description, WebGPU status, “Enter full-screen simulation”, optional Reset). Overlay is pointer-events-none so drag goes to the canvas; first click can also hide the overlay.

Design choices

R32Float for pressure/divergence keeps storage portable (some devices don’t support R16Float storage). No velocity diffusion in the pipeline yet—only advection, projection, and splat; viscosity is in params but not hooked into a diffusion solve. Advection uses nearest-neighbor sampling; bilinear or higher-order would improve quality later.

How to run it

npm run dev starts the Vite dev server; open in a WebGPU-capable browser (Chrome/Edge 113+). npm run build and npm run preview for a production build. Shaders live in shaders/ (WGSL); simulation logic is in src/simulation/.
